# 4. 本轮必须避免的十四个错误

## 错误 A：把 identity registry 存在当成 sovereign identity 已成立
registry 文件存在 ≠ 主权身份系统闭环。

## 错误 B：把 memory tiers 存在当成连续自我已成立
hot/warm/cold 存在 ≠ self continuity 已成立。

## 错误 C：继续添加字段但不统一语义
如果新增更多 `ownerId`, `identityId`, `memoryType`, `scope` 字段，却没有统一 contract，只会加复杂度。

## 错误 D：没有 canonical self object
如果系统没有明确“哪个对象是当前 runtime 的 canonical self record”，那么一切都不稳定。

## 错误 E：identity 与 memory 各写各的，不做统一收敛
本轮必须明确它们如何互相约束，不允许继续平行演化。

## 错误 F：SOUL 变成自由文本垃圾桶
SOUL / self narrative 不能变成无审计、无边界的情绪日志或营销文案。

## 错误 G：把任何重要记忆都默认长期保留
没有 salience / ownership / class boundary，长期记忆会失真并污染 self。

## 错误 H：没有 inheritance boundary 就谈 child / lineage
在未定义哪些记忆可继承前，不应把 lineage 当真实闭环。

## 错误 I：identity lifecycle 缺失
没有 rotation / revoke / recovery，就不能说 identity 是 durable sovereign self。

## 错误 J：只做 schema，不做 recovery / migration / verification
本轮必须证明：
- restarts work
- continuity verification works
- migrations or transitions are sane

## 错误 K：把 narrative update 当作模型自由发挥
自我叙事更新必须受治理和事实约束。

## 错误 L：没有测试就宣称自我连续性成立
identity-memory closure 必须可测试、可验证、可审计。

## 错误 M：把 user memory / self memory 混在一起
这会直接污染主体边界。

## 错误 N：过早做大量 UI 展示，而不是先做系统闭环
本轮优先级是 runtime truth，不是界面叙事。

# 6. 本轮必须完成的核心目标

你必须同时完成以下六类能力：

## A. 定义 Canonical Self Model

你必须收敛出一个统一的 **canonical self model**。至少包含：
- identity anchor
- continuity record / continuity chain position
- current active identity record
- current self-state
- wallet / signing relation
- soul / narrative reference
- validity / verification status

要求：
1. 必须明确“系统里什么对象代表当前 self”
2. 不能让 runtime 各处对 self 有不同解释
3. 必须提供稳定接口供 runtime / policy / memory / governance 使用
4. 必须可验证、可审计

## B. 建立 Durable Identity Registry & Lifecycle

至少完成：
1. persistent registry 语义收敛
2. identity record versioning
3. rotation / recovery / revoke 的基础 lifecycle contract
4. creator-authorized change boundary
5. canonical active identity resolution

要求：
- 不要只存更多字段
- 必须明确 record transition rules
- 必须能解释 identity 变更后 continuity 如何维持

## C. 建立 Identity-Aware Memory Contract

必须定义 memory ownership / memory class 体系，至少区分：
- self memory
- user memory
- environment memory
- session-only memory
- lineage-shareable memory
- non-inheritable memory

要求：
1. 明确哪些 memory 可以进入长期层
2. 明确哪些 memory 参与 self continuity
3. 明确哪些 memory 可被 child 继承
4. 不允许 self / user / environment 混写无边界

## D. 强化 Memory Consolidation 为 Self Continuity 管线

必须把 consolidation 从“记忆整理”推进成“自我连续性维护管线”的一部分。至少完成：
- session → episodic → semantic / relationship / procedural 路线的边界澄清
- salience / retention / forgetting 初步规则
- self-defining memory 的保留条件
- narrative-relevant memory 的更新触发条件

要求：
- 不能只增加摘要逻辑
- 必须说明哪些记忆影响 self narrative / identity continuity
- 必须避免长期记忆无限膨胀或被噪音污染

## E. 建立 SOUL / Narrative Update Governance

必须定义 SOUL / self narrative 的更新机制。至少回答：
- 什么事实有资格进入 self narrative
- 哪些是长期稳定身份事实
- 哪些只是 session noise
- narrative update 由谁触发
- narrative update 是否需要 verification / review / boundary checks

要求：
- 禁止让 SOUL 成为模型自由发挥文本
- Narrative update 必须受事实与治理约束
- 建议产出 structured update pipeline 或 ruleset

## F. 定义 Inheritance & Lineage Memory Boundary

为未来 replication / child runtime 做准备，必须先定义：
- 哪些 identity / memory 元素可继承
- 哪些不可继承
- lineage memory 的最小 contract
- child bootstrap 最少需要哪些 parent self facts

要求：
- 本轮不需要完整实现 replication
- 但必须为未来 replication 建立可信继承边界

# 7. 强制设计原则

你必须遵守这些设计原则：

## 7.1 先统一 self semantics，再扩张功能
不要继续加更多 identity 或 memory 功能，先收敛“什么是 self”。

## 7.2 先做 canonical model，再做周边集成
runtime、policy、memory、governance 都必须围绕 canonical self model，而不是各自定义 self。

## 7.3 narrative 不是装饰，是治理对象
SOUL / narrative 必须是可约束、可验证、可更新的对象，不是附属文案。

## 7.4 identity lifecycle 必须显式化
rotation / recovery / revoke 不能继续隐含在局部实现里。

## 7.5 memory ownership 必须制度化
长期记忆必须区分 self / user / environment / lineage 等 ownership / class。

## 7.6 为未来 economic / agenda / replication 提供稳定输入
本轮设计的 identity-memory contract，必须能成为后续三大工作包的稳定基础，而不是一次性 patch。
