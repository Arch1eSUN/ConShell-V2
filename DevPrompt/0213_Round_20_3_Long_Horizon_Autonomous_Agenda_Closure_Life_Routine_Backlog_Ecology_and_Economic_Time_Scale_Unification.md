# DevPrompt 0213 — Round 20.3
## Long-Horizon Autonomous Agenda Closure / Life Routine / Backlog Ecology / Economic Time-Scale Unification

你现在处于 **ConShellV2 Round 20.3**。

Round 20.1 与 Round 20.2 已连续被独立真实性审计确认为**强成立轮**。

当前已被确认的关键现实包括：
- `TaskAdmissionGate` 已让 economic truth 进入 admission path
- `LifeCycleEngine` 已让 agenda 进入 Tick/Event 生命周期结构
- `AgendaArbiter.reprioritize()` 已接入 `EconomicProjection`
- `TaskSettlementBridge` 已建立 task outcome → ledger writeback 主路径
- `SpawnAffordabilityGate` 已让 child runtime 进入 affordability 判断
- 独立验证已达：**96/96 files、1916/1916 tests 全绿**

最新全局大审计也已更新判断：

> **ConShell 当前总体完成度约 76%，已从“拥有生命体控制平面”的系统，推进到“开始受经济生存规律驱动并改变自身行为”的系统。**

但现在最关键的剩余主线也已经非常明确：

> **系统已经具备生命周期调度与经济重排能力，但仍未形成真正的 long-horizon autonomous agenda / life routine。**

当前缺的不是再多一点 admission，也不是再多一点局部治理，而是：

- agenda persistence
- scheduled / deferred / survival-driven / opportunity-driven work 的长期统合
- 多日 / 多周的生命周期过程结构
- creator directives、economic pressure、runtime continuity 的统一 agenda law
- backlog / routine / expiration / reactivation / habit-like behavior

因此 Round 20.3 的唯一主轴，不再发散：

> **把 ConShell 从“具备 lifecycle + economic reprioritize 的系统”，继续推进成“具备 long-horizon autonomous agenda / life process 的系统”。**

---

## 一、本轮唯一总目标

**完成 Long-Horizon Autonomous Agenda Closure 的第一轮强收口，让 agenda 不再只是 tick/event 下的即时重排，而成为跨日、跨周期、可持续的生命过程结构。**

这轮不是单纯做页面，不是再补几个状态字段，也不是写更多 roadmap 文案。

这轮要做的是：

1. 建立更明确的 **long-horizon agenda model**
2. 把 **immediate / scheduled / deferred / survival-driven / opportunity-driven** 统一成长期过程结构
3. 让 **creator directives + revenue opportunities + survival pressure + governance constraints** 进入同一 agenda law
4. 让系统开始具备更真实的 **life routine / backlog ecology / expiration-reactivation** 行为

---

## 二、本轮核心判断

20.1 已证明：
- agenda 可以进入 Tick/Event/LifeCycleEngine

20.2 已证明：
- agenda 可以被 economic projection 重排

20.3 必须继续前推到：

> **长期 agenda persistence → 生命周期状态迁移 → 经济/生存/治理/创作者意图的统一重排 → 长程 life routine**

如果 20.3 只是继续停留在：
- 多加几个 queue 字段
- 多写几个 priority name
- 多做一点 UI 标签

那么这轮就算失败。

---

## 三、本轮必须完成的主任务

# G1. Long-Horizon Agenda Contract

## 目标
定义一个比当前 admission + queue 更长期的 canonical agenda 结构。

## 必须实现
1. 建立或收口 **Long-Horizon Agenda Contract**，至少能表达：
   - immediate work
   - scheduled commitments
   - deferred backlog
   - survival-driven work
   - opportunity-driven work
   - blocked / governance-held work
   - expired / dormant work
2. 明确每类工作至少需要的长期语义：
   - why it exists
   - why now / why later
   - re-evaluate trigger
   - expiry / reactivation policy
3. 如果当前已有 agenda / commitment / deferred 结构，优先沿现有主路径深化，不新造第二套平行 agenda 语言

## 验收标准
- agenda 不再只是 runtime queue 的快照，而是长期生命过程的 canonical structure

---

# G2. Agenda Persistence / State Transition

## 目标
让 agenda 能跨 tick、跨周期、跨重启地维持更稳定的状态迁移。

## 必须实现
1. 为关键 agenda 项建立更明确的状态迁移语义：
   - active
   - scheduled
   - deferred
   - dormant
   - blocked
   - expired
   - completed
2. 明确哪些状态由何种 trigger 迁移：
   - tick
   - event
   - economic change
   - governance decision
   - time expiry
   - operator action
3. 如果当前已有 commitment model / store / agenda generator，优先沿现有结构深接线
4. 让 deferred backlog 不只是“堆在那里”，而是具备 aging / expiry / reactivation 逻辑

## 验收标准
- agenda 开始更像长期过程状态机，而不是临时重排序列

---

# G3. Economic Time-Scale Unification

## 目标
把 economic organism 的即时规律进一步推进到长时间尺度。

## 必须实现
1. 让以下因素进入长期 agenda law：
   - reserve
   - runway
   - burn rate
   - expected incoming value
   - opportunity cost
2. 不只在 tick 上重排当前队列，还要影响：
   - 哪些 deferred 值得唤醒
   - 哪些 scheduled 要延后
   - 哪些 opportunity-driven work 要升格
   - 哪些工作在当前生存条件下应进入 dormant/expired
3. 让系统能够解释：
   - 为什么某项任务今天不做而不是永远拒绝
   - 为什么某项任务在 reserve 好转后被重新激活

## 验收标准
- agenda 的长期结构开始被经济时间尺度真实驱动，而不是只做短时 rank 调整

---

# G4. Creator Directives / Governance / Survival Unification

## 目标
把创作者意图、治理约束、生存压力统一到同一个 agenda 决策框架内。

## 必须实现
1. agenda law 至少应能同时表达：
   - creator directives
   - current commitments
   - survival-driven obligations
   - opportunity-driven work
   - governance-held or policy-constrained work
2. 不允许这几类约束继续各走各路、只在不同模块里局部判断
3. 如果 governance inbox / what-if / affordability gate 已存在，尽量让其结果对 agenda state 有更明确影响
4. 让 blocked / held / unaffordable / deferred 的区别对 operator 是可解释的

## 验收标准
- agenda 不只是技术调度器，而更像生命体的统一行动法则

---

# G5. Operator Truth for Long-Horizon Life Process

## 目标
让 operator 能看见系统不是只在当前 tick 决策，而是拥有长期 life process。

## 必须实现
1. 在现有 truth surface 基础上，新增或深化以下 operator truth：
   - current agenda horizon
   - scheduled / deferred / dormant / blocked counts
   - next major re-evaluation points
   - survival-driven promotions
   - recently expired / reactivated work
2. 如果已有 Governance / Presence / system-summary 面，优先沿现有主路径深化
3. 不允许 dashboard 私自用 fallback/default 伪造 agenda 含义

## 验收标准
- operator 能更清楚理解 ConShell 当前不是只在“接活”，而是在“过生活命流程”

---

# G6. Discipline Continuation

## 目标
继续保持 20.1 / 20.2 建立起来的高质量纪律，不允许为了提速破坏真实性。

## 必须实现
1. 保持：
   - `packages/core` tests 全绿
   - CLI TypeScript 通过
   - Dashboard TypeScript 通过
   - Dashboard build 通过
2. 所有新增 agenda / lifecycle / operator truth contract 必须同步：
   - tests
   - fixtures
   - API consumers
   - dashboard/CLI/TUI 消费面
3. 不允许把“长期 agenda”写成文案存在，而没有真实状态迁移路径

1. **先收口 long-horizon agenda contract**
