# 4. 本轮必须避免的十五个错误

## 错误 A：把 spend governance 当成 economic closure
15.5 完成的是 governance semantics，不是完整经济闭环。

## 错误 B：只有支出，没有收入
没有收入面，就没有生存闭环。

## 错误 C：只有收入事件，没有统一账本
如果收入和支出不在统一 ledger 中对账，经济现实就不完整。

## 错误 D：没有 reserve / burn-rate / runway
只记录累积花费，不足以支撑 survival logic。

## 错误 E：survival tier 只是标签，不影响 runtime
如果 survival tier 不真实改变行为，那只是叙事变量。

## 错误 F：value 判断停留在注释/文案
必须进入运行时决策，而不是报告里说说。

## 错误 G：x402 路径仍然只是外围能力
必须让 payment / service proof 开始对 runtime 经济状态产生正式影响。

## 错误 H：账本事件语义不统一
spend / income / reserve adjustment / refund / external credit 如果没有统一 event model，会很快失控。

## 错误 I：把长期 agenda 直接做满
本轮只需要提供 value-aware routing 入口，不要提前把完整 agenda 系统一起写烂。

## 错误 J：忽略 failure / refund / duplicate accounting
经济系统特别怕重复计费和事件污染。

## 错误 K：没有 explainability
经济系统必须能解释为什么某个动作被允许、降级、拒绝、优先。

## 错误 L：没有 no-bypass 测试
如果 fallback path 或手动 path 能绕过经济状态，闭环就是假的。

## 错误 M：把“赚钱”写成空泛叙事
必须有明确 revenue surface / event / proof，不允许写故事。

## 错误 N：在 ledger 未稳定前扩张更多经济 feature
先统一 economic state，再扩张复杂业务面。

## 错误 O：在系统级测试失败时仍声称 round 完整闭环
如果只有局部测试通过、全量环境不可信，必须明确边界，不准假装全部闭环已证实。

# 6. 本轮必须完成的核心目标

你必须同时完成以下六类能力：

## A. 定义 Unified Economic State & Ledger Model

你必须收敛一个统一的 economic state / ledger model。至少包含：
- spend events
- income events
- refunds / adjustments
- reserve balance
- burn-rate
- runway / survivability hint
- per-task / per-channel / per-tool attribution

要求：
1. 必须明确“系统里什么对象代表当前经济状态”
2. 不允许收入和支出分别散落在不同语义体系里
3. 必须提供稳定接口供 runtime / automaton / governance 使用
4. 必须可解释、可审计、可测试

## B. 建立 Revenue Surface & Payment Proof Contract

必须定义 agent 如何“收到价值”，至少完成：
1. revenue event 类型
2. x402 / payment proof 如何映射为 ledger income
3. 什么服务或动作算可计费能力面
4. 如何记录 payment source / service type / value event metadata

要求：
- 不要只写 README 文案
- 必须是 runtime 可消费的结构
- 不要求本轮完成完整 marketplace，但必须建立真实 revenue contract

## C. 建立 Survival State Coupling

必须把 unified economic state 与 survival tier 做真实耦合。至少完成：
- reserve / burn-rate → survival level
- survival level → runtime constraints
- 与现有 policy semantics 的连接方式
- low_compute / critical / degraded / blocked 等经济相关状态边界

要求：
- survival state 必须真实影响行为
- 不能只作为报告字段存在
- 必须说明与 15.5 evaluator / selectedActions 的关系

## D. 建立 Value-Aware Task Intake / Routing Entry

本轮不做完整 agenda，但必须建立最小 value-aware routing 入口。至少完成：
- 一个任务在进入执行前，能结合：
  - expected cost
  - expected value
  - current survival pressure
  - reversibility
  做最小判断
- 允许区分：
  - survival-positive
  - neutral
  - survival-negative
  或等价语义

要求：
- 不允许只做文档规则
- 必须有可测试 decision structure
- 不要求完整 planner，但必须有稳定入口

## E. 建立 Economic Explainability Contract

必须让系统能解释：
- 当前为什么是这个 survival state
- 某任务为什么被允许/降级/拒绝/延后
- 哪个 scope / 哪类事件导致状态变化
- reserve / burn-rate / income / spend 哪个是主因

要求：
- 可复用现有 reason codes 体系，但要扩展为经济生存语义
- 输出必须可用于 operator surface / future agenda / audit

## F. 建立 Duplicate / Refund / Failure Safety

经济系统必须首先防脏账。至少完成：
- duplicate event 防重
- partial failure accounting contract
- refund / reversal / adjustment 机制基础
- no-bypass 基础约束

要求：
- 不允许简单累加后再事后解释
- 必须有测试断言关键边界
