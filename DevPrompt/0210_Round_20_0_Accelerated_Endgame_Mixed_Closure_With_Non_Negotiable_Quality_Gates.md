# DevPrompt 0210 — Round 20.0
## Accelerated Endgame / Mixed Closure / Non-Negotiable Quality Gates

你现在处于 **ConShellV2 Round 20.0**。

基于最新全局大审计（Round 19.9 基线），当前项目状态已经非常明确：

- ConShell 已经不是原型，不是概念工程，也不是只会讲故事的 agent monorepo
- 它已经完成了：
  - 高可信 runtime truth 基线
  - canonical control plane
  - Presence-first control surface
  - 初步 TUI / CLI / Dashboard / onboarding 一体化
  - agenda canonical truth surface
  - 初步 operator actionability
  - 87/87 files、1817/1817 tests 全绿
- 当前总体完成度约为：**66%**

但同样明确的是：

> **项目推进速度仍然偏慢，终局关键闭环还没有真正进入“最后冲刺”状态。**

当前最关键的剩余缺口已经收敛为：
1. **真实 economic closure**
2. **long-horizon autonomous agenda / life process**
3. **governed replication / collective evolution actualization**
4. **OpenClaw 级 session / tool / node / multi-surface orchestration 广度**
5. **release / performance / production hardening 的剩余尾债**

因此 Round 20.0 的战略不是单线推进，而是采用：

> **D. 混合推进（推荐方案）**
>
> - **70% 火力打终局主线**
> - **20% 火力打 OpenClaw 吸收**
> - **10% 火力打质量与发布硬化**
>
> 并且：**任何提速都不能以牺牲真实性、测试健康度、可审计性为代价。**

一句话：

> **Round 20.0 的目标是：显著提速，但不降质；在保持全绿和真实性纪律的前提下，把 ConShell 从“强平台”进一步推向“真正接近终局的自主生命体系统”。**

---

## 一、本轮唯一总目标

**用一轮高密度开发，同时推进终局生命主线、OpenClaw agent OS 吸收、以及必要的质量硬化，但必须保持强质量门槛与全绿纪律。**

本轮的核心不是“再做一点”，而是：

> **开始真正进入终局系统的加速阶段。**

但前提是：

> **加速 ≠ 降质；快 ≠ 粗糙；推进 ≠ 伪完成。**

---

## 二、20.0 的战略分配（硬约束）

## 70%：终局主线推进
必须优先打：
1. economic closure
2. long-horizon autonomous agenda
3. governed replication / collective evolution

## 20%：OpenClaw 能力吸收
必须优先打：
1. session orchestration
2. automation primitives / scheduled workflows / runtime control primitives
3. multi-surface / node-style runtime breadth

## 10%：质量与发布硬化
仅做高杠杆必要项：
1. verify-release 可信化
2. performance 尾债
3. operator workflow failure-path hardening

### 禁止事项
- 不允许 20.0 再被局部 polish 吞掉主火力
- 不允许为了“做得漂亮”牺牲终局主线推进
- 不允许为了赶速度把质量门槛降下来

---

## 三、本轮必须一次性完成的内容

# G1. Economic Closure 进入“真实生命规律”阶段

当前经济系统已经很强，但还不是终局闭环。

### 当前问题
现在已经有：
- settlement
- survival coupling
- profitability / gating
- economic truth surfaces

但仍没有完全做到：
> **真实价值创造 → 收入形成 → 资源状态改变 → 长期生存策略调整 → runtime 行为再分配**

### 本轮必须完成
1. 明确 ConShell 的**可出售能力面 / revenue surface**
2. 继续强化统一 economic ledger / reserve / burn / value accounting
3. 让 survival pressure 更强地进入：
   - model selection
   - task admission
   - agenda prioritization
   - background work allocation
4. 让 “earn your existence” 从理念更接近 runtime 主规律
5. 若已有 economic data 仍停留在 diagnostic 层，本轮继续往 behavioral law 推

### 验收标准
- economic state 不只是被显示，而是更实质地决定系统行为
- 至少一条真实 value → survival → behavior 链条更完整

---

# G2. Long-Horizon Autonomous Agenda

这是从“持续运行 runtime”走向“生命过程”的核心。

### 当前问题
现在已有：
- agenda truth
- scheduler / wake / scheduled autonomy
- continuity / restored work / priority hints

但还缺：
- 多日/多周 agenda persistence
- creator directives / opportunities / survival pressure / constitutional constraints 的统一计划结构
- 真正的 long-horizon autonomy

### 本轮必须完成
1. 把 agenda 从“当前态摘要”推进到“长期过程结构”
2. 让系统更明确地区分：
   - immediate tasks
   - scheduled commitments
   - deferred commitments
   - survival-driven work
   - opportunity-driven work
3. 改善 agenda 的 persistence / recovery / reprioritization
4. 让 agenda 的原因链更清楚：
   - 为什么现在做这个
   - 为什么延后那个
5. 如果存在 agenda contract 缺口，统一补齐 WebUI / TUI / CLI 可消费面

### 验收标准
- agenda 更像长期生命计划的一部分，而不是短期状态快照

---

# G3. Governed Replication / Collective Evolution 从骨架走向真实闭环

当前 lineage / collective / multiagent 仍然偏骨架，不足以代表终局生命系统。

### 本轮必须完成
1. 推进 child / lineage / collective runtime 的真实执行闭环
2. 推进 funding / reporting / recall / governance policy 的真实连通
3. 明确 replication 不只是“能生成”，而是：
   - 何时允许生成
   - 生成后如何受治理
   - 如何跟踪健康 / 成本 / 产出
4. 如果已有 multiagent 结构还只是 manager object，本轮优先推进 actualization
5. collective truth surface 与 operator control 面要更清晰显示真实 collective state

### 验收标准
- lineage/collective 从“结构存在”向“真实系统行为闭环”明显推进

---

# G4. OpenClaw Session / Tool / Runtime Orchestration 吸收提速

这部分必须开始提速，但不能抢掉终局主线火力。

### 本轮必须完成
优先吸收 OpenClaw 的高杠杆部分：
1. **session orchestration model**
   - 更清晰的 main / isolated / child / operator-facing session semantics
2. **automation primitives**
   - scheduled flows / wake / trigger / webhook-like runtime semantics（按仓内真实方向落地）
3. **multi-surface runtime cohesion**
   - WebUI / TUI / CLI / runtime actions 更强对齐
4. **tool runtime elevation**
   - 让关键工具真正成为 first-class runtime primitives，而不是零散 helpers

### 验收标准
- OpenClaw 向的吸收不是只体现在命名，而是体现在 runtime orchestration 深度上

---

# G5. Truth → Action 闭环继续增强

19.8 / 19.9 已经开始做 operator actionability，但还远不够。

### 本轮必须完成
1. 至少让一组关键 truth surface 达到更强闭环：
   - visible
   - canonical
   - explainable
   - actionable
   - auditable
2. 提升 doctor / export / inspect / diagnostics 的 operator-facing 整合
3. 让 operator 不只是跳页面，而是真能对关键问题采取下一步动作
4. 保持所有动作在治理与风险边界内

### 验收标准
- control plane 更像“生命体控制面”，而不是“只读看板”

---

# G6. Release / Verification Hardening（只做必要高杠杆项）

当前 release discipline 已有文档，但自动验证脚本不可靠。

### 本轮必须完成
1. 修复 `scripts/verify-release.sh` 的 false negative 问题
2. 确保 release gates 与真实仓内验证口径一致
3. 若还存在脆弱 gate，继续硬化
4. 不追求华丽，只追求：
   - 可信
   - 可重复
   - 可签收

### 验收标准
- “green means green” 不再被脚本误判破坏

---

# G7. Performance 尾债治理（只清高杠杆残留）

性能还没终局收口，但本轮不允许它吞掉主线火力。

### 本轮必须完成
1. 继续治理明显 >500kB 的大 chunk
2. 优先处理 wallet / heavy vendor / non-critical surface 的加载代价
3. 保持 canonical structure 不被破坏
4. 若无法在本轮完全解决，必须留下更精确的残余债说明

### 验收标准
- 性能继续改善，或至少性能债边界更清楚

## 四、本轮强制执行纪律

### D1. 速度必须提升，但不能用“跳过验证”换速度
本轮必须保持：
- core tests 全绿
- CLI tsc 通过
- Dashboard tsc + build 通过

### D2. 任何新增 canonical contract，必须同步 tests / fixtures / consumers
不允许重演 19.8 的 contract 改了、测试没跟上的问题。

### D3. 不允许用 UI 漂亮掩盖生命主线没推进
如果本轮主要成果只是 UI polish，这就是失败。

### D4. 不允许把 OpenClaw 吸收变成广度炫耀
优先吸收 high-leverage system principles，不追求表面矩阵扩张。

### D5. 不允许为了推进 replication / autonomy 而放松治理边界
powerful actions 仍必须受 constitutional / governance / audit 约束。

### D6. 不能口头宣称完成，必须以仓内验证为准
任何“完成”都必须可被独立复核。

> **在保持高可信质量门槛的前提下，显著提速推进 ConShell 的终局生命主线。**
