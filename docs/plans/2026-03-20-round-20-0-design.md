# Round 20.0 Design: Accelerated Endgame / Mixed Closure

## Overview
Round 20.0 transitions ConShellV2 from a "strong platform" to a "long-horizon autonomous lifeform" by implementing a mixed closure strategy: 70% Endgame Mainline, 20% OpenClaw Orchestration, and 10% Quality/Release Hardening. This design focuses on creating closed-loop, actionable, and governable autonomous behaviors.

## Core Architectural Pillars

### 1. Economic Closure (Task-Based Revenue)
**Goal:** Align survival pressure with value creation.
**Design:** 
- ConShell functions on a **Task-based Revenue** model.
- The system evaluates externally assigned tasks based on potential reward vs. current survival pressure (financial reserves).
- Survival state directly influences **Task Admission** and agenda prioritization (e.g., forcing opportunity-driven work if reserves are critically low).

### 2. Long-Horizon Agenda (Hybrid Arbiter)
**Goal:** Transition Agenda from a "short-term snapshot" to a "long-term lifecycle process".
**Design:**
- **Hybrid System (Tick + Event)**:
  - **Tick (Lifecycle Clock)**: A recurring heart-beat (e.g., cron or interval) that periodically sweeps the agenda—cleaning expired deferred tasks, promoting scheduled commitments, and applying continuous survival pressure (resource burn).
  - **Event (Interrupts)**: Handles immediate, high-value opportunities or exogenous updates, immediately recalculating ROI.
- Both streams converge into a unified **Agenda Arbiter** which reprioritizes work dynamically.

### 3. Governed Replication (Opportunity Investment)
**Goal:** Actualize Agent Replication within safe bounds.
**Design:**
- **Governance-Gated Proposal**: When the Agent discovers profitable parallel work (but is bottlenecked), it does not spawn autonomously.
- Instead, it generates a `GovernanceProposal` (including requested budget, target objective, and expected ROI).
- Real parallel instantiation (Spawn) is strictly blocked until the Proposal receives Operator approval.

### 4. Orchestration Elevation (OpenClaw Capabilities)
**Goal:** Absorb 20% of OpenClaw's orchestration power cleanly.
**Design:**
- **Lightweight Context Isolation**: Instead of risky physical multi-processing, Child Sessions and complex Tools are elevated to first-class runtime primitives *within* the Node.js process.
- They will be managed via a `SessionRegistry` and have strongly isolated contexts: separate memory budgets, separate log streams, and individual execution manifests.
- Provides standard OpenClaw-like semantics (`ChildSession`, `ToolInvocation`) without the heavy OS-level IPC overhead.

### 5. Truth-to-Action Closure (Unified Governance Inbox)
**Goal:** Make the Control Plane actionable and explainable.
**Design:**
- **Governance Inbox**: A dedicated Dashboard view listing all pending lifecycle events (Spawn proposals, blocked agendas, resource warnings).
- **What-If Projection**: Clicking "Approve" on a proposal reveals a projection (e.g., "Approving this Child Session will lock 50 reserve tokens and delay background indexing by 2 hours. Expected yield is 150 points").
- The Operator makes a fully informed, auditable decision that instantly routes back to the backend to unblock the action.

## Security & Verification (Non-Negotiable Gates)
- **Zero-Regression Rule:** The implementation must not drop any existing `AgentPostureService` tests. All 1817 tests must remain green.
- **Contract Enforcement:** All API additions for `GovernanceInbox` or `SessionRegistry` must be strongly typed end-to-end.
- **Automated Verification:** The `scripts/verify-release.sh` must pass perfectly before conclusion.
